Day 1 · Foundations

What is an agent? Three evolving definitions

Each definition layers on the previous one.

Classic

LLM + tool

The LLM reasons in language; the tool performs the action.

Newer

LLM + context

Behavior is driven by what you feed the model — context engineering.

Current

LLM + harness

The orchestration scaffolding: retries, tool routing, middleware.

Keep the terms straight: OpenAI = model provider · GPT / Gemini / DeepSeek / Kimi K2 = LLMs · ChatGPT = an application built on GPT.
Day 1 · Foundations

The core split

Each half supplies exactly what the other lacks.

LLM

  • Understands natural language
  • Reasons about what should happen
  • Cannot perform any action — no search, no DB call, no file write

Tool

  • Just code — a function, method, or API call
  • Performs real actions
  • Cannot understand natural language
Agent = the two combined. The LLM decides; the application-side tool executes.
Day 1 · Foundations

Six shortcomings of raw LLMs

The reasons agents and tools exist at all.

Hallucination

Plausible-sounding but wrong output.

Knowledge cutoff

No awareness after the training date — can’t answer “who won IPL 2026.”

Stateless

Every request is fresh. “Memory” is the app resending prior context.

Fixed context window

Input is limited — and every token costs money.

Non-deterministic

The same prompt phrases its answer differently every time — a reliability concern.

No actions

It only generates text. It can’t search, query, or write anything.

Day 1 · RAG

What can still go wrong

These four problems motivate every RAG variant in the rest of the course.

Wrong tool choice

The LLM picks a tool that can’t answer the query.

Too many tools

Every declared tool consumes context on every call.

Oversized responses

Chunking code is harder than chunking prose

Naive chunking breaks code

Day 2 · Chunking

Chunk overlap: don’t split the meaning

Chunk A“…this is a 5”
✂
Chunk B“week course…”

Neither chunk contains “5 week course” — the phrase is destroyed at the boundary.

The fix

Let adjacent chunks share trailing/leading text, so any split phrase survives intact in at least one chunk.

No universal value: short-sentence data needs little overlap; long sentences need more.

Day 2 · AST

AST — how machines actually read code

Compilers, linters, and IDE autocomplete never read code as plain text.

  • A tree of the code’s real structure — a function is a node with name, args, body; a class contains its functions as children
  • Python stdlib: ast.parse() builds it · ast.dump() prints it · ast.walk() traverses · nodes map back to exact source
  • Production choice: tree-sitter with per-language grammars (tree-sitter-python)
Day 2 · AST

AST chunking vs recursive splitter

Only the chunking step changes — embedding, storage, retrieval stay identical.

Recursive splitter

  • Pattern-matches on class / def tokens
  • Falls back to newline / character splits
  • No understanding of symbol relationships

AST chunking

  • Parses the whole file into a tree first
  • One chunk per function / class / method — real units
  • Full control over boundaries while walking the tree
Trade-off: the tree costs memory. Keep the AST only transiently to extract chunks, then discard — persist just chunks + embeddings.
Day 2 · AST

AST in practice: wins and limits

The Day-1 failure, fixed

With AST chunks + k = 4, “what are the functions inside auth service” returns a clean, complete answer.

Not a universal fix

  • k = 1 (large-codebase simulation) still failed: ~22 s, ~2.8k tokens on retries
  • Cross-file queries (“every place billing error is raised”) — one chunk = one function, never a codebase view
  • Built per file · code-only · still needs docstrings
Cross-file relationships need a graph representation → Graph RAG (Day 3).

Day 2 · Lexical

When “similar” is wrong and “identical” is right

Semantic search wastes exact-match queries

Searching order_handler semantically also returns order_processor and order_manager — noise you pay for.

And grep doesn’t scale

Raw text scanning across thousands of files is too slow. Elasticsearch-style engines solve this with an inverted index.

(Though Claude Code deliberately greps live — Day 4 tells that story.)

Day 2 · Lexical

The inverted index

Forward index

document → its content

How you normally read.

Inverted index

word → [docs containing it] + freq

Hash-map-like, near O(1) lookup per term.

Build steps

  • Tokenize the documents
  • Drop stop words (a, the, in, on, for…) — zero retrieval value
  • Map each word → documents + frequency

Trade-off: storage-heavy, but buys much lower query-time latency — classic space-vs-speed.

Day 2 · Lexical

BM25 — best matching 25

Inverted index + relevance scoring. The default inside Elasticsearch. Three factors:

1 · Term frequency

More occurrences = more relevant — with diminishing returns: 200 hits isn’t twice as relevant as 100.

2 · Word rarity (IDF)

“agent” in an AI corpus discriminates nothing; rare words like “RAG” weigh far more.

3 · Length penalty

A term counts for more in a 10-word doc than a 1,000-word one — long docs accumulate hits cheaply.

Core weakness — purely lexical: querying “generate bill” scores zero against a genuinely relevant “create invoice” document. No shared words → no match. No synonyms, no paraphrase.
Day 2 · Lexical

Semantic vs lexical — when each wins

Semantic wins

  • Natural-language questions
  • Paraphrased / conceptual queries
  • “Explain why my payments are failing”

Lexical / BM25 wins

  • Exact exception or error-code lookups
  • Exact class / function / variable names
  • Anything where identical beats similar
Context rot: BM25’s length-penalty intuition applies to LLMs too — a short, focused context gets proportionally more attention; a huge one dilutes it and the model loses the query. Relevance degrades as context grows. Don’t stuff context just because it fits.

Day 2 · Combining

Agentic RAG — first look

An extra LLM call, purely for decision-making.

Hybrid

Always calls a fixed set of retrievers → merge → re-rank. Deterministic, engineered ahead of time.

Agentic

An LLM classifies the query and picks retriever(s) per request — possibly just one, saving retrievals.

Cost trade-off: the decision call adds latency and money. If both retrievers should always run, hard-code the hybrid call — agentic RAG earns its cost only when strategy must vary per query.

Day 3 · Validation RAGs

Validating the output: self vs corrective

Pass retrieved context + query + generated answer to a model and ask: “is this good?”

Self-RAG

Another call to the same LLM / session, asking it to verify its own response.

Corrective RAG (CRAG)

Validation by a different model — or the same LLM in a fresh session with no shared context.

Honest caveats

An extra check ≠ a guarantee of correctness. And every validation call adds latency + cost.

Aliases

Reflection RAG, fact-verification RAG — all the same idea: one extra validation layer.

Day 3 · Taxonomy

RAG is not just vector DBs

The retriever can pull from any store — and every database is optimized for a different kind of query.

SQL RAG

LLM writes the SQL against your existing DB — no migration needed.

Web RAG

Live web search as the retriever.

Graph RAG

Relationships: who calls what, what depends on what.

Files / PDFs

Plain documents, HTML, anything readable.

Tool descriptions must carry the schema — put your data structure in the tool’s docstring so the LLM can generate valid queries against it.
Day 3 · Graph RAG

Code is inherently a graph

Killer queries

  • “Who calls this function?”
  • “Trace the complete call chain”
  • “If I change this, what breaks?” — multi-hop impact analysis

The cost anecdote

The instructor’s live “trace the complete call chain” query — run via hybrid RAG with no call limit — burned an enormous number of tokens in one call.

A graph DB answers it trivially and cheaply.

AST is per-file, local chunking. The graph captures cross-module relationships — classes, inheritance, calls: basically UML, and UML is a graph.
Day 3 · Graph RAG

How Graph RAG works

The store

  • Neo4j — used in the final project (in-memory graphs are demo-only)
  • Nodes: classes, functions, modules · Edges: calls, imports, has-method
  • Demo graph: AuthService, User, TaskService… with typed relationships + properties

Who writes the graph query?

Not you — the LLM writes the Cypher. You send it the tools plus a description of your graph structure (nodes, relationships); it returns the query; your app executes it.

Same pattern as every other retriever.

Day 3 · Architecture

The final-project storage design

One codebase, indexed three ways — each store answers what it’s best at.

Vector DB

Chroma-like · semantic meaning: “how does auth work?”

BM25 / Elasticsearch

Inverted index · exact matches: error codes, function & class names

Neo4j

Graph · relationships: call chains, imports, impact analysis

Storage duplication is real — the same corpus lives three times. That’s the quality-vs-storage trade-off, accepted deliberately. (And not the only valid design — you can skip the graph with metadata tricks.)
Day 3 · Transformations

Fixing bad queries and bloated chunks

HyDE — query transformation

A big reason RAGs fail: user queries are bad — vague, under-specified. HyDE has the LLM first generate a hypothetical answer, then uses that richer text to drive retrieval.

Fun fact: even passing the same query twice can improve RAG performance.

RAPTOR — chunk transformation

Too many chunks, unsatisfying retrieval? Summarize the chunks and run RAG over the summaries.

Both get full treatment on Day 5.

Day 3 · Retrieval metrics

Precision & recall

Precision — correctness

relevant retrieved ÷ total retrieved

“Of everything you fetched, how much was relevant?” Low precision = noise sent to the LLM.

Retrieved 5, relevant 2 → 40%

Recall — completeness

relevant retrieved ÷ total relevant in corpus

“Of all relevant info that exists, how much did you get?”

200 docs, 10 relevant; retrieved 5, of which 3 relevant → P = 60%, R = 30%

Day 3 · Retrieval metrics

The precision–recall tug of war

1,000 docs in the corpus, 100 of them relevant:

Fetch all 1,000

Recall = 100% · Precision = 10%

Perfect completeness, terrible noise: latency, cost, hallucination risk, the LLM drowns.

Fetch exactly 1 relevant doc

Precision = 100% · Recall = 1%

Perfectly clean, hopelessly incomplete — unacceptable in medical or legal domains, where recall wins.

Both at 100% is mathematically possible (retrieve exactly the relevant set) — realistically, pushing one up pushes the other down.
Day 3 · Retrieval metrics

@K and the ranking metrics

You only send top-K chunks to the LLM (typically 3, 5, or 10) — so measure at K.

Precision@K

relevant in top-K ÷ K

Recall@K

relevant in top-K ÷ all relevant

Hit rate

Binary: 1 if any relevant chunk retrieved, else 0.

NDCG

Graded relevance — not all relevant docs are equally relevant; weights by rank.

MRR — mean reciprocal rank

Position of the first relevant doc → take 1/rank → average over queries. First-relevant at positions 4, 3, 1, 4 → MRR = mean(¼, ⅓, 1, ¼). If MRR is already good, you may be able to skip re-ranking entirely.

Day 3 · Demo

Computing the metrics — what K reveals

10-doc knowledge base, golden dataset of query → relevant doc IDs, run at K = 1, 3, 5.

Day 4 · Ranking

Three levels of ranking

1 · Retrieval ranking

Inside the store. Vector DBs rank by cosine/dot product; lexical stores by BM25/TF-IDF. You never implement this.

2 · Fusion

App side, no LLM. Merge multiple ranked lists purely by rank arithmetic — standard algorithm: RRF. Never re-reads documents.

3 · Re-ranking

App side, with a model. Cross-encoders, ColBERT, LLM re-rankers re-read the candidates and re-order by true relevance.

None of these stages is mandatory — skip fusion, skip re-ranking, use any subset. It’s a design decision measured by evals.

Day 4 · Hybrid code

Hybrid in code: EnsembleRetriever

Setup

  • Dense retriever (HuggingFace or OpenAI embeddings, k = 2)
  • BM25Retriever over the same chunks
  • LangChain EnsembleRetriever(weights=[0.5, 0.5]) — internally uses RRF

Tunable dimensions

Which retrievers · chunk size & strategy · K per retriever · fusion weights · re-ranking on/off — every combination shifts the eval metrics, so measure.

Also: add the middleware call limits in code, not just on the LangSmith side — prevents runaway loops in every environment.

Deliberately uses no RAG index — it greps/globs the repo live, every time.

Day 5 · RRF

RRF — reciprocal rank fusion, internals

The formula

score(doc) = Σ 1 / (k + rank)

  • Rank 1 matters more than 2 → use the reciprocal
  • But 1/1 vs 1/2 is too steep — rank 2 isn’t “half as good”
  • Smoothing constant k = 60 (convention): 1/61, 1/62, 1/70…

Why it’s loved

  • Considers rank and cross-retriever agreement
  • No training, no LLM call — pure arithmetic
  • Works across dense, sparse, graph — anything that ranks
Demo: “How does the app enforce task limits per user?” — a chunk appearing in both lexical and semantic lists (create task) scored ≈ 0.03 vs ≈ 0.016 for single-list chunks. Agreement boosts a chunk.
Day 5 · Agentic RAG

Agentic RAG — the full picture

Any RAG pipeline where an extra LLM call makes a decision.

Routing

Which store — vector, Mongo, SQL, one of several vector DBs?

Validation

Post-retrieval “is this relevant?” loops — self/corrective RAG qualify as agentic.

Decomposition

Multi-step queries split into sub-queries by an extra call.

Retrieve at all?

The LLM may decide no retrieval is needed and answer directly.

Agentic vs hybrid (interview classic): hybrid = all retrievers always fire, then fuse. Agentic = an LLM decides which fires. If the router picks multiple → it’s both. Production systems combine agentic + hybrid + corrective.

Day 5 · HyDE

HyDE — hypothetical document embeddings

Paper: “Precise Zero-Shot Dense Retrieval without Relevance Labels.”

Vague query“how does auth work?”
→
LLM writes a hypothetical answerfake but plausible passage
→
Embed the hypotheticallands in a richer neighborhood
→
Retrieve real docsby similarity to it
→
Answer from real docsnever from the fake one

Why it works

Users write abstract, under-specified queries — you can’t force them to rephrase. The hypothetical document carries far more retrieval signal than the raw query.

Trade-offs

Can hallucinate, can occasionally hurt accuracy, adds token cost. Poor fit for code (too codebase-specific). Make it conditional — a decision call “is this query good enough?” → that combo is agentic RAG.

Day 5 · RAPTOR

RAPTOR — tree-organized retrieval

Recursive Abstractive Processing for Tree-Organized Retrieval. Problem: too many chunks → search is slow.

  • Build: recursively cluster chunks by embedding; an LLM writes a summary per cluster; parents hold child summaries — repeat upward
  • Search: semantic search against summaries, pick best cluster per level, drill down to leaf chunks
  • Store: anything tree-friendly — SQLite works, graph DBs too
  • vs Graph RAG: RAPTOR groups & summarizes (trees, no cycles); graphs model relationships (cycles allowed)
Day 5 · Vectorless

PageIndex — vectorless, reasoning-based RAG

The #1 trending GitHub repo that day. Core claim: “similarity does not mean relevance.”

The mental model: a book’s table of contents

Looking up SQL? You go to the Databases chapter — you reason from titles, you don’t compute similarity.

  • Build: LLM summarizes each file/section → directory → file → chunk tree (demo: 34 chunks, zero embeddings)
  • Query: send TOC + query to the LLM → it reasons “probably section 3” → descend → leaf = the answer document

vs RAPTOR — know this cold

Both build trees. RAPTOR traverses by semantic search over summaries; PageIndex uses no embeddings at all — pure LLM reasoning over the index, like a B+-tree traversal.

Wins: standard structured documents — legal docs full of cross-references. Loses: concepts scattered across distant sections, where semantic search shines. Build cost: LLM summarization of the corpus, once.

Skip fusion and re-rank directly? Possible — but you send far more docs to the model. Fuse first (math, cheap), re-rank the survivors (model, expensive). Fusion is the first interview round; re-ranking is the final.

Everything you pass the LLM per call is context. It’s limited, and every token costs money. Why manage it explicitly? One-word interview answer: LLMs are stateless.

Day 5 · Context engineering

Lost in the middle

Paper: “Lost in the Middle: How Language Models Use Long Contexts.” Order matters.

Primacy effect

Beginnings are privileged — training data opens with instructions, headings, intros. Learned bias.

Recency effect

Ends are privileged — next-token prediction attends most to recent tokens. Architectural bias.

Practical rule: put key instructions, the query, and critical facts at the start or end of the context — never buried in the middle.
Day 5 · Context engineering

The five context operations

Expansion

Deliberately add context: HyDE; fetching a function plus its class and helpers.

Compression

Too many tokens? Summarize. Combine with expansion: pull everything, then shrink it.

Selection

Choose a subset without summarizing — fusion/re-ranking picking 10 of 100 is selection.

Pruning

Clean it: dedupe overlapping retriever results, drop stale outputs and irrelevant data.

Ordering

Reorder so important pieces sit at the beginning and end — exploit primacy/recency.
