Ch 03

Objects → Vectors: Embeddings kya hote hain

Core idea: kisi bhi object (document, image, query) ko numbers ki ek list mein badal do — aise numbers jo uska "meaning" capture karein. Is list ko vector ya embedding kehte hain.

Intuition ke liye 2D se shuru karo. Har document ko do properties pe plot karo: x-axis = length (kitna lamba hai), y-axis = kitna real/non-fictional hai. Toh:

  • Harry Potter — lamba + fictional → right-bottom.
  • Sherlock Holmes — lamba, thoda believable → right, Harry Potter se upar.
  • "Attention Is All You Need" paper — chhota + 100% real → left-top.

Jaise-jaise aur documents plot karoge, clusters banenge: textbooks ek jagah, short stories ek jagah, journals ek jagah. Matlab: position = meaning. Document ka meaning ek topological/spatial space mein aa gaya.

2 dimensions kaafi nahi

Duniya ke saare documents describe karne ke liye 2 properties kam hain. Popularity ke liye teesri dimension, publish year ke liye chauthi... aise toh infinite dimensions chahiye honge, jo impossible hai. Isliye hum dimensions ko ek fixed number d pe constrain karte hain — jaise Llama mein d = 4096. Bade models aur zyada dimensions use karte hain.

Ab sawaal: infinitely complex object ko fixed d dimensions mein kaise map karein? Answer: neural network. Millions of documents feed karo, network deep patterns seekh leta hai. Input = text document, Output = 1D array of size 4096 — yehi vector hai. Similar objects → paas-paas ke vectors.

Do vectors "similar" kab hain? Jab unke beech ka distance kam ho. Common measures:

Model ko sikhaya kaise jaata hai

Training mein hum model ko reward/penalize karte hain. Example: ek object age=89 (senior) aur doosra age=21 (young) — ye bahut different hain, toh inke vectors door hone chahiye. Agar model ne paas rakh diya → penalize (loss badhta hai, backpropagation se weights change hote hain). Door rakha → reward. Iterate karte-karte network achha ho jaata hai. Isse vector loss functions pe training kehte hain (contrastive learning family).

Fine-tuning: apne use case ke liye

Mostly ready-made embedding libraries/models kaafi hain — pre-trained, well-tested. Par kabhi-kabhi customize karna padta hai. T-shirt website ka image search example: off-the-shelf model photo mein model ka face, complexion, posture sab vector mein daal dega — jo tumhare liye noise hai. Tumhe sirf t-shirt se matlab hai.

  • Solution: network ko examples dikhao jahan expression/posture change hota hai par output same rehta hai → model irrelevant cheezein ignore karna seekh jaata hai.
  • Cost: manual data banana + GPU training — time aur paisa lagta hai. Cloud providers ya khud TensorFlow/PyTorch se kar sakte ho.
Yaad rakhne wali baatVector = distilled meaning. Raw image/document bada aur noisy hota hai; hum use compress kar ke aakhri layer ka 1D array nikaalte hain jo sirf matlab ki cheezein rakhta hai. Similar objects → similar (paas-paas) vectors.

Ch 05

Vector DB hi kyun? SQL/NoSQL kyun nahi?

Natural sawaal: relational databases already hain, unhi mein store kar lo na? Problem ye hai:

  • Index 1D ke liye bane hain. Naam pe index lagao → sorted data → binary search → fast. Par 4096 dimensions pe binary search kaise karoge? Data format hi suit nahi karta.
  • ACID aur fast updates ki zaroorat nahi. Relational DBs transactions guarantee karte hain — vector search ko iski zaroorat hi nahi. Wo overhead waste hai.

Vector databases do cheezon mein optimized hain:

  1. Compression: vectors ko efficiently compress kar ke store karna.
  2. Fast n-dimensional search: clustering se nearby vectors jaldi dhoondhna — poore cluster ko ek centroid point se represent kar dena.
Trade-off dhyan rakhoVector DBs exactness ki guarantee nahi dete. Financial systems (transactions, exact retrieval) ke liye ye bilkul galat choice hai. Ye search, recommendations, text-generation jaise inexact use cases ke liye hain — thodi accuracy chhod ke bahut speed milti hai.

Ch 07

Vectors store kaise hote hain: Compression, Clustering, Quantization

Naive search = query ko har document se compare karo = O(n). Scale pe ye bahut slow hai. Do techniques bachaati hain:

1. Compression (Scalar Quantization)

Har dimension normally 2 bytes (16 bits) leti hai. Usse ghata ke sirf 2 bits kar do — values ko buckets mein baant do:

0–25 → 00
26–50 → 01
51–75 → 10
76–100 → 11
16 bits → 2 bits per dimension ≈ ~90% memory saving. Distinct vectors bhi kam ho jaate hain → search chhoti space mein hoti hai → fast.

2. Clustering

Saare vectors ko clusters (clumps) mein baanto. Har cluster ka ek centroid vector hota hai. Query aane par saare vectors se compare karne ke bajaye sirf centroids se compare karo, sahi cluster dhoondo, phir uske andar nearest documents dekho.

In dono ka mix — pehle compress, phir compressed vectors ko cluster karna — vector DB world mein quantization (product quantization) kehlata hai. Memory footprint aur search time dono aur ghat jaate hain.

DrawbackCompression mein information loss hota hai → perfect similarity match nahi milega. Clustering mein sabse relevant document galti se doosre cluster mein ja sakta hai → wo consider hi nahi hoga. Par scale pe ye trade-off worth hai: thodi accuracy ke badle zabardast speed.

Ch 08

Search Algorithms: Quad tree se HNSW tak

Relational DB wala indexing (ek column sort + binary search) yahan fail ho jaata hai kyunki distance saari dimensions milakar nikalti hai, kisi ek column se nahi. Toh kaunse algorithms kaam karte hain? Timeline dekho:

Quad tree → KD-tree

  • Quad tree: 2D space ko recursively 4 regions mein todta hai. Query ke liye har level pe decide karo — top-left/top-right/bottom-left/bottom-right — jab tak ek vector na bache. Sirf 2D ke liye.
  • KD-tree: quad tree ka k-dimensional version. Har level pe koi ek dimension choose kar ke split karo (e.g. dim1 < 3 left, ≥ 3 right), neeche jaate raho, aakhri cluster mein brute-force. Problem: space bahut leta hai, high dimensions mein accuracy/performance kharab (curse of dimensionality).

ANNOY (Spotify)

Approximate Nearest Neighbors Oh Yeah. Idea simple: do random points lo, space ko aadhe mein kaat do. Repeat karte raho jab tak clusters chhote na ho jaayein. Query aane par decision tree se neeche jao, apne cluster mein search karo. Expected O(log n), par practice mein performance average — Spotify khud 2022 mein isse hat ke HNSW pe chala gaya.

HNSW — aaj ka standard

Hierarchical Navigable Small World. Naam lamba, idea simple — skip list jaisa layered graph:

  1. Level 0: saare vectors, har vector apne k nearest neighbors se connected. Kuch "popular" vectors (jinse bahut log jude hain) hote hain.
  2. High-degree (popular) vectors ko upar wale level pe promote karo (thoda randomness ke saath). Repeat — upar jaate-jaate kam, representative vectors bachte hain.
  3. Search: top layer se shuru karo. Current layer mein koi aur vector query ke zyada paas hai? Wahan jump karo. Nahi? Ek layer neeche utro. Neeche jaate-jaate space detailed hoti jaati hai, par tumne har layer pe dher saare vectors skip kar diye → O(log n).
Upar ki layers = express highways (bade jumps), neeche ki layers = local galiyan (exact search). Har level pe zyada-tar vectors skip ho jaate hain.

IVF — Inverted File Index

Very large scale ke liye: k-means se document vectors ko clusters mein baanto, har cluster ka centroid rakho. Query → nearest centroid(s) dhoondo → sirf unhi clusters mein search karo. Parameter k (kitne clusters search karne hain) tunable hai — zyada k = zyada accurate, zyada slow. Drawback: ho sakta hai centroid door ho par uske cluster mein tumhara asli nearest vector ho — wo miss ho jaayega.

Big pictureYe sab Approximate Nearest Neighbor (ANN) algorithms hain — approximation se accuracy thodi girti hai, par memory aur speed ka fayda bahut bada hai. Production mein aaj sabse common: HNSW (pgvector, Qdrant, Weaviate sab mein) aur IVF+PQ (billions-scale, FAISS/Milvus).

Case Study: Milvus ki internal architecture

Milvus ek popular open-source vector DB hai (~32k+ GitHub stars, kaafi companies use karti hain). Iski architecture dekh ke samajh aata hai ki scalable vector DB ke engineering decisions kaise dikhte hain. Write ka safar:

Teen coordinators: Root (DDL — schema/table changes), Data (insert/update/delete), Query (fetch). Har request apne type ke coordinator ke paas jaati hai.

Andar kya chal raha hai

  • Data nodes: writes lete hain; data periodically compact ho kar files (S3/HDFS) mein jaata hai — sorted string tables jaisa.
  • Index nodes: in files se index banate hain (HNSW ya tumhara chosen algorithm) — index files ke cache jaise.
  • Query nodes: sirf data fetch pe focused — ek cache ki tarah, Kafka se constantly update hote rehte hain.
  • Kafka cluster: writes ke events data nodes aur query nodes dono ko pahunchata hai — persistence + real-time updates.
  • Consistency: default eventual (mostly kaafi hai); strong consistency ka option bhi hai — data persist hone ke baad hi query nodes padh sakte hain.

Achhi baat: separation of concerns — har cheez independently scale hoti hai (writes ke liye data nodes badhao, reads ke liye query nodes). Criticism (course instructor ki bhi): har coordinator ek single point of failure hai — ye design smell hai. Baaki sab ek normal distributed database jaisa hi hai.

TakeawayVector DB mein koi magic nahi hai — wahi load balancers, proxies, message queues, compaction, caching. Fark sirf itna: store hone wala data vectors hai aur indexes HNSW/IVF jaise ANN structures hain. (Ye chapter skip-able hai agar system design basics nahi aate.)

Ch 11

Tokenization: Text ko tukdon mein todna

Model raw text nahi padhta. Pehle text ko tokens mein toda jaata hai — words ke tukde. catching = catch + ing, balling = ball + ing. Kabhi-kabhi "George R.R. Martin" jaisa naam ek hi token ban jaata hai. Har token ko ek unique ID milti hai — in IDs ka set model ki vocabulary hai.

Flow: text → tokens → IDs → model → output IDs → decode → text.

Raw text seedha kyun nahi dete?

  • Compression: character-by-character padhne se sequence bahut lambi ho jaati hai. Attention ka cost O(n²) hai jahan n = sequence ke tokens ki count — toh jitne kam tokens, utna sasta.
  • Meaning: catch token ka apna inherent matlab hai (cricket ho ya fishing), ing ka matlab hai "abhi ho raha hai". Tokens meaning ke natural units hain.
  • Separation of concerns: tokenization (encoding/decoding) aur LLM alag-alag rakhe jaate hain, taaki tokenizer independently improve ho sake. Algorithms: Byte Pair Encoding (BPE) sabse common; Meta ka Byte Latent Transformer naya research hai jo tokenizer hata ke bytes pe kaam karta hai.

Ch 12

Training: Model seekhta kaise hai (aur kitna mehenga hai)

Training data = high-quality sources: arXiv research papers, GitHub ka open-source code, Wikipedia. Web crawlers data laate hain, HTML/PDF parse hota hai, tokenize hota hai, aur phir:

  1. Model ko sentence do: "I went to the mall".
  2. "I" ke baad kya aayega? Model probabilities deta hai: "went" 60%, "am" 20%...
  3. Sahi predict kiya → reward; galat → penalize (backpropagation se weights adjust).
  4. Trillions of tokens pe ye repeat karo → model sentences ki structure mein expert ban jaata hai.

Kharcha kitna?

  • DeepSeek (V3, 2024-25): claim ~$5.5M sirf GPU/electricity cost ka. Llama: isi order ke millions. Ye sirf compute cost hai — hardware (GPUs/TPUs) acquire karna alag se bahut bada investment hai.
  • Frontier models (GPT-4 class aur aage) ka असली total cost $100M+ estimate hota hai.
Practical adviceStartup ya mid-size company ho? Apna foundational model mat banao. Ye apna database banane jitna (ya usse zyada) mushkil hai. Existing models use karo — open-weight (Llama, DeepSeek-R1, Qwen, Mistral) ya APIs (OpenAI, Anthropic, Google).

Ch 14

Accuracy badhao: Self-validation, System Prompt, Few-shot

Engineer ke naate teen cheezein chahiye: accurate responses, kam cost, achha UX. Pehle accuracy — hallucination rokne ke teen hathiyaar:

1. Self-validation

Model se answer generate karwao, phir usi se poochho: "kya ye sahi hai?" Interesting fact: models evaluate karne mein generate karne se behtar hote hain — jaise insaan ke liye painting banana mushkil hai par painting achhi hai ya nahi, batana asaan. Answer dene se pehle ye extra check laga do.

2. System Prompt (Prompt Engineering)

Model ko batao wo kaun hai: "Tum ek customer support executive ho..." Ye system prompt hai (user prompt = actual sawaal). LLMs general trained hote hain — koi bhi personality le sakte hain. Personality restrict karo → response quality upar.

3. Few-shot Prompting

Query ke saath similar solved examples bhi bhejo — "delivery late ho toh kya bolte hain" ke 2-3 real Q&A pairs. Model ko format + behaviour dono samajh aa jaata hai. (Zero-shot = koi example nahi; few-shot = kuch examples.)

UX aur Safety: Context, Agents ka intro, PII protection

Accuracy ≠ UX. Accuracy = sahi hona; UX = user ko wahi dena jo wo dhoondh raha hai. User poochhe "is t-shirt ka size kya hai?" — असली sawaal hai "mujhe fit hogi kya?". Ch 17

Attention ka intuition: "Crane" wala example

Word crane ka matlab kya hai? Akele mein — pata nahi. Par sentence mein turant clear ho jaata hai:

  • "The crane damaged its wings while flying" → पक्षी (bird) 🐦
  • "The crane has an arm made of steel" → machine 🏗️
  • "They had to crane their neck" → body posture

Tumne matlab kaise nikala? Aas-paas ke words dekh ke. Yehi kaam attention mechanism karta hai. Problem ye hai: tokenizer dono sentences mein "crane" ko same token ID → same starting vector deta hai. Context vector mein nahi hota.

Attention kya karta hai

  1. Sentence ke saare words ka ek n×n similarity matrix banao (har word ka har word se score — cosine similarity jaisa).
  2. "crane" ke liye scores: the→0, crane→1, damaged→0.5, wings→high, flying→high.
  3. Ab crane ke vector mein weighted push do: crane + 0.5×wings + 0.5×flying → crane ka vector bird-meaning ki taraf khisak jaata hai.

Result: har word ke vector mein uske sentence ka contextual information embed ho gaya. Sirf crane nahi — "arm" bhi crane ke context mein "metal arm" ban jaata hai, human arm nahi.

Ch 18

Attention ka math: Q, K, V, Softmax, Temperature

Ab wahi cheez mathematically. Matrix ke rows = queries (Q) — "main kis word ke liye attention dekh raha hoon", aur columns = keys (K) — "wo word mujhe kitna affect karta hai".

  1. n words, har word ka d-dimension vector → Q matrix hai n×d, K bhi n×d.
  2. Q × Kᵀ (n×d · d×n) = n×n similarity matrix.
  3. Scale down: √d se divide karo — warna values ke bade hills/valleys ban jaate hain, training unstable hoti hai.
  4. Softmax: values ko exponential se smooth probabilities mein badlo (har row ka sum 1).
  5. Is attention matrix ko V (values, n×d) se multiply karo → har word ka vector "sahi force" se push hota hai → n context-aware vectors.

Attention(Q,K,V) = softmax(QKᵀ/√d)·V — "Attention Is All You Need" (2017) ka famous formula.

Ek nahi, stack of transformers

Attention ke baad ye vectors ek chhote feed-forward network (2 layers) se guzarte hain jo next word predict karta hai. Galat prediction → backpropagation → weights update. Aur ye poora block (attention + FFN) ek transformer layer hai — models mein aise 12 se 96+ layers stack hote hain. Har layer ka output agli layer ka input. Aakhri layer ke embeddings pe linear function chala ke sabse likely next token ki probability milti hai.

Greedy vs Temperature

Ch 19

Transformer architecture detail: GPT = Decoder-only + Masking

"Attention Is All You Need" paper ke famous diagram mein left side encoder hai, right side decoder. GPT jaisi models purely decoder architecture use karti hain — encoder wala hissa aur beech ka cross-attention block hata do, bacha: masked multi-head attention + feed-forward network.

Masked attention: training ka jugaad

Sentence: "my family member passed away". Model ko sikhaana hai ki har prefix ke baad kya aata hai:

  • "my" → "family" predict karo
  • "my family" → "member" predict karo
  • "my family member" → "passed" predict karo ... aur aage bhi

Ek-ek prefix alag se bhejna bahut slow hoga. Trick: poora sentence ek saath bhejo, par attention matrix ka upper half mask (zero) kar do — har word sirf apne se pehle ke words dekh sakta hai, future nahi. Ek hi pass mein saare prefixes ki predictions parallel mein train ho jaati hain.

Ch 20

Multi-Head Attention: Ek cheez ko kai nazariye se dekhna

Ek embedding dimension socho: size. Hydrogen atom = 0.00000001, apple = 10cm, black hole = trillions. Ek hi scale pe itna variation handle karna mushkil hai. Space ki baat karte waqt planets wale scale mein sochna hai; atoms ki baat mein nano scale mein.

Solution: linear transformations (computer graphics jaisa — image ko stretch/shrink/translate karna). Attention formula mein beech mein weight matrices ghusa do:

  • Q × W_q (n×d · d×k) aur K × W_k — ye matrices space ko alag "perspective" mein transform karti hain.
  • V ke liye bhi apna W_v hai.
  • Har aisa perspective = ek attention head. Kai heads = multi-head attention.

Ek head "space-scale" cheezon pe focus kar sakta hai, doosra grammar pe, teesra atoms pe... Sab heads ke outputs ko concat + combine kar ke ek final output banta hai. Aur best part: ye weight matrices bhi trainable hain — galti hui toh sirf FFN nahi, W_q/W_k/W_v ke values bhi backpropagation se sudharte hain. Zyada data = behtar transformations.

Head h ... apna-apna lens

Ch 21

KV Caching: Dubara mat gino jo gin chuke ho

Models mein 300 billion+ parameters hote hain (Q/K/V weight matrices + FFN weights milakar; naye models mein trillions). Training aur inference dono optimize karne padte hain. Pehla, sabse standard optimization: KV (Key-Value) caching.

Input: "the crane damaged its wings while" (6 tokens) → 6×6 attention matrix bana. Model ne "flying" generate kiya. Ab 7 tokens ke liye 7×7 matrix chahiye. Par dhyan se dekho: pehle 6×6 wala hissa bilkul same rahega — purane words ka aapas ka context badla nahi. Naya sirf: aakhri row aur aakhri column.

Ch 22

Paged Attention: KV cache ki memory ka OS-style solution

KV cache ki ek badi problem: pata nahi kitna bada hoga. "I am going to the restaurant..." ka answer 4 words ka bhi ho sakta hai, 40 ka bhi. Production mein lakhon concurrent users hain, limited GPU memory hai. Ab:

  • Over-allocate karo (har user ko 1GB) → memory waste (naive implementation mein 40–60% memory waste hoti thi).
  • Under-allocate karo → cache chhota pad gaya → recompute karna padega.

Solution OS se udhaar liya: Paged Attention — KV cache ko fixed-size blocks (pages) mein todo, jaise CPU virtual memory. Virtual blocks logically continuous dikhte hain, par physical blocks memory mein kahin bhi bikhre ho sakte hain. Cache dynamically grow hota hai → ~100% memory utilization → zyada users, zyada throughput.

Bonus: Prefix sharing

Reasoning models ek prompt se multiple continuations banate hain ("...because I wanted to meet a friend" / "...because I was hungry"). Sab ka prefix same hai — toh prefix ka attention block physical memory mein ek hi copy rakho, saare parallel branches usi ko share karein. Memory aur bachi.

Ch 24

Flash Attention: GPU ko samajh ke likha gaya attention

GPU ke paas do memory hoti hain: SRAM (chip pe, chhoti par bijli-fast) aur HBM/DRAM (badi par slow). Vanilla attention har step (matmul → softmax → matmul) pe slow memory se data laata-le jaata hai — ye load/store hi bottleneck hai, almost IO call jaisa.

Flash Attention ka idea: attention ko chhote blocks (tiles) mein todo, ek block SRAM mein lao, saare operations (matmul + softmax + matmul) wahin fuse karke ek saath karo, sirf final result wapas bhejo. Mathematical tricks (online softmax) se ye blocks baad mein sahi tarike se aggregate ho jaate hain.

Ch 25

Model Quantization: Weights ko chhota karo

Model ki memory = parameters ki count × har parameter ke bits. Weights normally 32/16-bit floats hote hain. Unhe 8-bit ya 4-bit mein represent karo → 50–90% memory saving + faster inference (zyada weights cache mein fit hote hain). Drawback: accuracy thodi girti hai, especially 4-bit/2-bit pe.

Smart tareeka: Train full, serve quantized

  • Training: high-precision weights (accuracy critical hai, backprop sensitive hai).
  • Inference: training ke baad quantize karo (post-training quantization) — 21 din training + 1 din quantization.
  • Static quantization: saare weights same bits mein.
  • Dynamic/mixed quantization: sirf wo weights quantize karo jinka loss pe minimum impact hai; sensitive weights high-precision rahein. Better quality, par complex aur savings kam.

Ye Ch 07 wale vector-compression jaisa hi idea hai — wahan stored vectors compress kiye the, yahan model ke khud ke weights. GPT series, Llama series — sab quantization use karte hain. (Local LLM chalane wale GGUF/4-bit models isi ke products hain.)

Ch 26

Sparse Attention: Har word ko sab kuch dekhne ki zaroorat nahi

Attention O(n²) hai — core algorithm pe hi vaar karte hain. Observation: kisi word ka context usually uske paas hi hota hai. Harry Potter gussa hua toh wajah 3 sentences pehle hogi, 6 pages pehle nahi (mostly).

Toh n×n matrix ke bajaye ek thick diagonal compute karo — har token sirf apne ±w neighbors dekhe. Cost: O(n·w). Drawback: door ka context miss ho jaayega → accuracy thodi giri.

Google BigBird ka clever mix

  • Local attention: thick diagonal (paas ke tokens).
  • Global attention: kuch important tokens (jaise [CLS]) ki poori row/column compute karo — wo sabko dekhte hain, sab unko.
  • Random attention: kuch random cells bhi — door ka context milne ke chances badh jaate hain. Tunable: zyada random = zyada accurate = zyada cost.
BigBird = local + global + random. Orange: random cells ko b×b blocks mein rakha jaata hai — GPU blocks mein data process karta hai, bikhri hui 1×1 cells bhejne ka IO cost poore block jitna hi hota, toh block-sparse hi practical hai.
2026 UpdateSparse attention pehle widely used nahi thi, par DeepSeek ka Native Sparse Attention (NSA, 2025) aur isi family ke GPU-centric designs ne ise practical bana diya — lambi context windows (1M+ tokens) isi direction se aa rahi hain.
Ch 27

Deployment: Sahi size ka model + Distillation

Ab theory se practice. Production mein pehla sawaal: model kitna bada ho? Ek reality check:

  • Llama-7B jaisa "chhota" model ~50–100 concurrent queries handle karta hai per NVIDIA A100 (~$3/hour).
  • 10 GPUs = $30/hour ≈ $20,000+/month. Startup ke liye bhaari.

Cost ghataane ke raaste: quantization (Ch 25), vector dimensions kam karna, FFN parameters kam karna — aur sabse powerful: Small Language Models (SLMs). 100B → 100M parameters = 1000× chhota. Intelligence poori transfer nahi hoti (lossy compression), par support queries / search jaise use cases ke liye kaafi hoti hai.

Distillation: Teacher-Student

  1. Teacher = bada LLM. Student = chhota model jo teacher ko mimic karna seekh raha hai.
  2. Dono ko same input do: "my eyes hurt after staring at the ___". Teacher: "screen". Student: "pigeon" 😅. Difference backpropagate karo.
  3. Naive tareeka kharab hai: sirf top-token compare karoge toh student unfairly punish hoga (uska 2nd guess "screen" ho sakta tha). Isliye poori probability distribution (soft targets) compare hoti hai — teacher ke saare token probabilities ke kareeb aana kaafi hai.

(Ye GANs ki yaad dilaata hai — wahan generator vs discriminator ka competition hai; yahan student teacher ko copy karta hai.) Isliye har bada model family apne chhote distilled versions bhi release karti hai (Llama 70B → 8B, DeepSeek-R1 → distilled Qwen versions, waghera).

Ch 29

Reinforcement Learning: Chess se seekho

LLM word-probabilities pe trained hai — usse "mate in 3 dhoondo" bologe toh wo random/illegal moves dega, kyunki wo game ki physics nahi samajhta, bas "is move ke baad kaunsa move aata hai" jaanta hai. Chess games pe train karne se bhi novel best move nahi milega.

Chahiye: Reinforcement Learning (RL) — model ek environment mein hai, observations dekhta hai, actions leta hai, aur har action pe reward milta hai (achha move +, bura move −). Lakhs of games khelte-khelte model duniya ka representation bana leta hai. Google DeepMind ka AlphaZero isi se bana — aaj ke sabse strong chess engines ki foundation.

Ch 30

RLHF aur PPO: Insaan reward dete hain

Essay ke liye reward kaun dega? Insaan. Isliye naam: Reinforcement Learning from Human Feedback (RLHF). Model 5 blogs generate karta hai, human batata hai kaunsa pasand hai → wo direction reinforce hoti hai. Practical dikkat-ein:

  • Scoring mushkil hai, ranking asaan: "100 mein se kitne?" nahi, "kaunsa behtar?" — isliye ChatGPT type apps tumse do responses mein se ek choose karwate hain.
  • Feedback mehenga aur rare hai: log answer chahte hain, ratings nahi dete; kuch randomly/maliciously click karte hain.

PPO: Sparse feedback se continuous space

Proximal Policy Optimization. "Policy" = function: state do, action batata hai. Humans millions points pe feedback nahi de sakte, toh jahan +100 ka real feedback hai uske aas-paas ki space ko +99, +98... se bhar do (diminishing returns ke saath) — sparse points se ek smooth, continuous, differentiable reward space ban jaati hai jisme gradient descent chal sakta hai. Result: model waise answers dene lagta hai jo insaano ko pasand aate hain.

Dhyan doYe abhi reasoning nahi hai — ye bas "humans ko khush karna" seekhna hai. Par ye har reasoning model ki foundation hai. (2026 tak PPO ke saath DPO aur DeepSeek ka GRPO bhi standard alternatives ban gaye hain — same goal, asaan training.)
Ch 31

Chain of Thought: Sochne ke tokens

Chain of Thought (CoT) = model answer se pehle thoughts (intermediate sentences) generate karta hai. "100 ÷ 200?" → "common factor 100 nikaalta hoon... 100/100=1... 200/100=2... toh 1/2 = 0.5". Lamba lagta hai, par:

  • Fine-tuning ke waqt model ko <thought>...</thought> tags wale structured examples pe train karte hain — "ye user ko mat dikhao, internally use karo".
  • Zyada thought tokens = zyada compute per question = behtar answers. Complex task → lambi chain; simple task → chhoti.
  • RL upar lagao → model sahi answers ki taraf nudge hota hai.

Self-consistency

Ek thought ke bajaye kai variations generate karo (thought1, thought1', thought1''...), aur majority/centroid wala chuno (thoughts ko vectors bana ke centroid ke sabse paas wala). Complex reasoning tasks pe kaafi improvement.

Common confusionCoT prompt engineering nahi hai. Prompt mein "step by step socho" likhna sirf output pe constraint hai. Asli CoT fine-tuning/training time pe sikhaayi jaati hai — model khud thoughts banana seekhta hai. (2026 tak: OpenAI o-series, DeepSeek-R1, Claude extended thinking, Gemini thinking — sab isi principle ke production versions hain.)

Ch 35

AI Agents: LLM ke upar ka wrapper jo kaam karta hai

Definition: software jo independently intelligent tasks perform kare jo business requirements meet karein. Example — AI travel agent: "Bali trip plan karo, book karo, manage karo" → wo research karega, bookings karega, notifications pe react karega (email bhejna, booking update karna).

Andar se agent = LLM (brain) + MCP clients (external systems se baat) + tools (calculator, currency converter...) + apna system prompt + memory.

Agents ke 3 fayde

  1. Isolation of concerns: booking agent alag, travel-manager agent alag, feedback/upsell agent alag — har ek ka apna LLM, apne tools, apni team. Microservices jaisa organization benefit.
  2. Feedback se improvement: "budget cross kar diya" jaisi user feedback agent ke behaviour mein aati hai.
